### 二重評価の正当化

実行されても何も検出されない検出器は、*実行できない*または*スキャン対象が何もない*検出器とは根本的に異なります。「scanned: 0」という値は、このイベント全体で最も警戒すべきシグナルです。つまり、次のいずれかに当てはまるということです。

-トークンストア/データベースにアクセスできませんでした
-ロジックエラーまたはスキーマの不一致により、クエリが行を返しませんでした
-タイムウィンドウフィルターにより、評価前にすべての候補が除外されました
-検出パイプライン自体が壊れている

**実稼働環境の SOC 環境では、「scanned: 0」の検出器は、そうでないことが証明されるまで、潜在的な死角として扱う必要があります。**

## 2。考えられる説明を可能性別にランク付け

### `スキャン済み:0`状態の場合 (主な懸念事項)

#### ランク 1 — 良性:評価期間にトークンが存在しない* (可能性:高) *
>スキャンの時点で、システムにはしきい値ウィンドウ内にアクティブな JWT 発行がない可能性があります。これは、ピーク時以外やメンテナンスの時間帯、またはデプロイの初期段階でも運用上有効です。

**これを裏付ける証拠:** EventBridge Schedulerの呼び出しは時間ベースです。アクティブなユーザーがいない状態で午前3時にスキャンを実行した場合、レコードはゼロになると予想されます。

**リスク:** 確認されれば低い。検証なしで想定した場合は高い。

---

#### ランク 2 — 運用:データソース接続またはクエリ障害 * (可能性:中～高) *
>ディテクタがトークンストア (Redis、DynamoDB、RDS など) への接続に失敗したか、クエリの設定ミスが原因で空の結果セットが返されたか、「scanned`」が生成される前に飲み込まれたサイレント例外が発生した可能性があります。

**これを裏付ける証拠:** `findings_total: 0`と`scanned: 0`の組み合わせは疑わしい。健全なシステムであっても、データパイプラインが壊れていない限り、*一部の*トークンが発行され評価されているはずだ。

**悪用された場合の攻撃経路:** 検出パイプラインが壊れていることを知った攻撃者は、警告を発することなくトークンを発行したり備蓄したりできる無期限の時間枠が与えられます。

---
#### ランク 3 — 構成のずれ:しきい値またはフィルターの設定ミス* (発生率:中) *
>ディテクター設定の「threshold_minutes: 5」の値が、トークンのTTLまたは発行タイムスタンプと一致していない可能性があります。5 分以内にトークンが評価されたり、タイムスタンプの比較にタイムゾーンや形式のバグがあったりすると、スキャン前にすべてのレコードが除外されてしまいます。

**バグパターンの例:**
```Python
# 不正解:ナイーブな日時とタイムゾーン対応の日時の比較
issued_at = トークン_レコード ["issued_at"] # ナイーブ UTC 文字列
しきい値 = datetime.now ()-タイムデルタ (分 = 5) # ローカルタイム
# サーバーが UTC でない場合、この比較ではすべてのレコードが自動的に除外されます。
issued_at < 閾値の場合:
 スキャン候補.追記 (トークンレコード)
```

---

#### ランク 4 — 敵対的:検出器抑制またはログ操作* (発生率:低) *
>検出パイプライン、EventBridge ルール、またはトークンストアへの書き込みアクセス権を持つ巧妙な攻撃者が、検出器の出力を抑制する可能性があります。これは発生頻度は低いものの、影響は大きいシナリオです。

**注目すべき指標:**
-ディテクターの実行ロールに影響する最近の IAM ポリシーの変更
-クラウドトレイルのイベントブリッジルール変更
-Lambda 関数コードの変更または環境変数の改ざん
-異常な「PutEvents」または「DeleteRule」の API 呼び出し

---

#### ランク5 — 脅威シナリオ：トークン・ハーベスティング/クレデンシャル・スタッフィングの前兆*（発生可能性：この特定のイベントでは低いが、戦略的リスクは高い）*
> >実際の未使用のトークンが存在する場合、最も懸念される説明は、攻撃者が (認証情報の乱用、フィッシング、またはセッションハイジャックによって) 有効なJWTを取得するために認証を行い、その後そのJWTを意図的にすぐに使用しなかったというものです。これは、行動分析のトリガーを回避するため、または後で別のIP/デバイスから使用して位置情報ベースの制御を回避するためです。

**攻撃チェーン:**
```
1。攻撃者が有効な認証情報を入手する (フィッシング、クレデンシャルスタッフィング、内部関係者)
2.攻撃者の認証 → JWT の発行 → ログ記録
3.攻撃者はトークンをすぐには使用しない (速度ベースの検出を回避)
4.C2 インフラストラクチャにトークンが流出
5.異なる地域/デバイスから数時間/数日後に使用されたトークン
6.この時点で、「未使用トークン」の警告ウィンドウは終了しています。
```

**Blast radius: ** JWT のクレームに付与されたすべての権限によるアカウントの完全乗っ取り。トークンの適用範囲が上限 (管理、書き込み、請求) の場合、その影響は重大です。
