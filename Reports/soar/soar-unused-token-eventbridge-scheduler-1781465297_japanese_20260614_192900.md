## 2。考えられる説明を可能性別にランク付け

### ランク 1 — 良性:ユーザーがセッションを放棄した (最も可能性が高い ~ 45%)
ユーザーが認証 (ログインリンクのクリック、SSO の開始など) した後、ブラウザーを閉じたり、別の場所に移動したり、何らかのアクションを実行する前に中断されたりした。これは、実稼働環境で最も統計的に一般的な説明です。

**サポート指標:**
-異常な認証信号なし (MFA 障害なし、異常な IP なし)
-シングルトークンの発行 (トークンの更新試行なし)
-ダウンストリーム API 呼び出しは記録されません

**リスク:** 単独ではごくわずかです。同じユーザー/アカウントでこのパターンが繰り返される場合は昇格します。

---

### ランク 2 — 良性:自動ヘルスチェックまたは統合テスト (~ 20%)
CI/CD パイプライン、合成監視ツール、または統合テストスイートでは、トークンをビジネスロジックに使用せずに認証情報の有効性を検証することがよくあります。

**サポート指標:**
-認証タイムスタンプが導入期間またはスケジュールされたジョブと一致している
-サービスアカウントまたは人以外の ID が関係している
-既知の自動化ユーザーエージェントに発行されたトークン

**リスク:** 低い。ただし、サービスアカウントが未使用のトークンを大規模に発行すると、トークンストアが膨張し、失効管理の課題が生じる可能性があります。

---

### ランク 3 — オペレーショナル:ディテクタパイプライン障害 (~ 15%)
「scanned: 0」という結果は、ディテクタ自体がトークンストアを列挙できなかったことを強く示唆しています。これは次のことを意味する可能性があります。
-ディテクタが読み取り権限を持たないデータストアにトークンが発行された
-スケジューラーが実行される前にトークンの有効期限が切れていたか、削除されていた
-バグ、設定ミス、または競合状態により、トークンストアクエリが空の結果を返しました
-EventBridge スケジューラーは起動したが、ダウンストリームの Lambda/Worker は通知なしで失敗した
**リスク:** 中～高。検出器が壊れると、系統的な死角が生じます。このウィンドウで発行されたトークンは、良性であれ悪意のものであれ、検出されません。

---

### ランク 4 — 疑わしい:クレデンシャルスタッフィング/アカウントプロービング (~ 10%)
有効な認証情報 (フィッシング、情報漏えいデータベース、またはパスワードスプレーを通じて取得) を持つ攻撃者は、認証を行って認証情報の有効性を確認し、実際のリソースアクセスに関連する行動分析が行われないようにセッションを放棄します。

**攻撃経路:**
```
攻撃者が認証情報を取得 (侵害/フィッシング)
 → 認証を行って有効性を確認する (「認証情報検証プローブ」)
 → トークンを使用しない (データアクセスアラートの発生を防ぐ)
 → 確認済みの有効な認証情報を後で使用できるように保存する
 → 後で新しい認証を行って戻り、実際の攻撃を行う
```

**これを高める裏付けとなる指標:**
-通常とは異なる位置情報や ASN からの認証
-営業時間外の認証
-ユーザーには最近のログイン履歴がありません
-複数のアカウントが同じパターンを同時に表示している
-自動化ツール (curl、python-requests など) に関連するユーザーエージェント文字列

**リスク:** 確認されれば中～高。攻撃者は有効な認証情報を確認したので、戻ってくる可能性があります。

---

### ランク 5 — 疑わしい:トークン流出の試み (~ 7%)
部分的なシステムアクセス権限を持つ攻撃者（SSRF、内部サービスの侵害など）は、認証フローをトリガーしてトークンを発行させ、通常の API パスを経由するのではなく、発行メカニズム（トークンストア、ログ、ネットワークトラフィック）からトークンを収集しようとします。

**攻撃経路:**
```
攻撃者は SSRF または侵害されたサービスを介して認証フローをトリガーします
 → JWT を発行してトークンストアに書き込んだ/応答
 → 攻撃者は帯域外でストレージ (DB、Redis、ログ) からトークンを読み取る
 → トークンは通常の API パスで「使用されたことはない」が、すでに侵害されている
```

**リスク:** 確認されれば高い。このトークンは、ディテクタからは見えないチャネルを通じて攻撃者によって積極的に使用されている可能性があります。

---

### ランク 6 — 悪意のある:偵察/タイミング攻撃 (~ 3%)
巧妙な攻撃者は、トークンをリソースへのアクセスに使用することを意図せずに、トークンの形式、有効期限、署名アルゴリズム、または応答タイミングを観察して、認証システムのフィンガープリントを認証します。

**リスク:** 攻撃者が大規模なキャンペーンのためにシステムをマッピングしている場合、確率は低いが影響は大きい。

## 3。アナリストが推奨するアクション

### 即時 (0 ～ 1 時間)

**ステップ 1: 最初に検出器の障害を優先順位付け**
「scanned: 0」という結果が、最もアクションにつながる差し迫った懸念事項です。何もスキャンしない検出器は誤った保証となります。

```
アクション:検出器の状態を確認
-EventBridge の実行ログでスケジューラーが起動していないか確認してください

-ワーカーの実行ログにエラー、タイムアウト、権限の失敗がないかを確認する
-ディテクターのトークンストア読み取りアクセスの IAM 権限を検証します
-直近 60 分以内に発行されたトークンをトークンストアに手動で照会する
```

**ステップ 2: 認証イベントの関連付け**
```
アクション:認証コンテキストをすべて取得する
-送信元 IP アドレスと位置情報
-ユーザーエージェント文字列
-ユーザーアカウントタイプ (ヒューマンアカウントとサービスアカウント)
-認証方法 (パスワード、SSO、MFA、API キー)
-認証時間 (営業時間?週末?珍しい?)
-アカウントの過去の認証パターン
-このアカウントが以前にこのパターンを示したことがあるかどうか
```

**ステップ 3: 帯域外トークンの使用状況を確認**
```
アクション:通常の API パス以外でのトークンの使用状況を検索
-特定の JWT のアプリケーションログのクエリ (存在する場合は jti クレームによる)
-任意のログアグリゲーションシステム (SIEM、CloudWatch、Splunk) でトークンを検索する
-エラーログ (トークンで失敗したリクエスト) にトークンが表示されていないか確認する
-認証されたユーザーの IP からの接続がないか、ネットワークフローログを確認する
```
