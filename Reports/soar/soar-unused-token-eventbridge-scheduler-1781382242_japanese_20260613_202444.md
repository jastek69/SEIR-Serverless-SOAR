## 2。考えられる説明を可能性別にランク付け

### ランク 1 — 良性のユーザー行動 (確率:~ 55%)
**説明:** ユーザーは認証されたが、何らかのアクションを実行する前にセッションを中止した。一般的な原因には以下が含まれます。
-ログイン直後にブラウザタブが閉じられる
-低速ネットワークで、ユーザーが操作できるようになる前にセッションがタイムアウトする
-ユーザーが認証されて内容を確認し、他の場所で回答が見つかり、タブを閉じた
-トークン使用前の OS によるモバイルアプリのバックグラウンド

**エビデンスの調整:** 単一イベント、相関異常なし、横方向の動き、リソースへのアクセス試行なし

**リスク:** 最小限。標準ユーザー行動パターン。

---

### ランク 2 — 自動スクリプトまたはボット認証プローブ (確率:~ 20%)
**説明:** 認証された自動プロセスで、トークンを消費せずに認証情報を検証したり、エンドポイントの可用性をテストしたりできます。これには次のような特徴があります。
-クレデンシャル・スタッフィング攻撃に使用される**クレデンシャル・バリデーション・ボット** — 認証して有効性を確認し、認証情報を後で使用できるように保存する。現在のセッションでは絶対にトークンを使用しない
-ログインフローをテストするがダウンストリーム API は実行しない**合成監視ツール**
-**CI/CD パイプラインの設定ミス** 認証は成功するが、後続の API 呼び出しステップは通知なしで失敗する

**攻撃経路:**
```
攻撃者が認証情報リスト (侵害データベース) を取得
 → 自動ボットがターゲットに対して認証を行う
 → 認証成功 = 認証情報が有効であることが確認されました
 → トークンは破棄されました (まだ必要ありません)
 → 後で標的型攻撃に備えて認証情報が保存される
 → 数日/数週間後:アカウントの完全乗っ取り
```

**Blast Radius: ** これがクレデンシャル・スタッフィングの場合、ブラスト範囲は収集されたリストのすべてのアカウントに及びます。未使用のトークンは*ノイズの少ない確認ステップ*で、実際のダメージは後で発生します。

**検出ギャップ:** 従来のブルートフォース検知 (ログイン失敗のしきい値) では、**認証が成功する**という理由で、このパターンに全く当てはまりません。未使用のトークン検出が存在するのはまさにこのためです。ただし、これは「scanned > 0」の場合に限られます。

---
### ランク3 — トークン・ハーベスティング/マン・イン・ザ・ミドル・インターセプト (確率:~ 10%)
**説明:** 攻撃者が送信中 (または XSS、ログインジェクション、または安全でないストレージ経由) に JWT トークンを傍受し、後で使用できるように保持しています。正当なユーザーは正常に認証されましたが、トークンは暗黙のうちに漏洩しました。

**攻撃経路:**
```
正規ユーザー認証 → JWT 発行
 → 攻撃者がトークンを傍受する (XSS /ネットワーク傍受/ログ公開)
 → 正規のユーザーセッションが終了する（アプリの観点からは「未使用」のトークン）
 → 攻撃者は後で別の IP/デバイスからトークンを使用する
 → トークンは有効期間が長い場合でも有効です
```

**主なリスク要因:** 有効期限が長い (15 分を超える) JWT トークンは、「未使用」の検出期間が終了した後もずっと悪用可能です。トークンのTTLが1時間または24時間であれば、攻撃者は悪用できる期間が広くなります。

**検出ギャップ:** 未使用のトークン・ディテクターは使用されていない場合は検出しますが、閾値ウィンドウを過ぎると*遅延*使用は検出されません。しきい値を超えたトークンを別の IP/デバイスフィンガープリントから使用するには、別の動作検出ルールが必要です。

---

### ランク 4 — アプリケーションエラーまたは統合エラー (確率:~ 10%)
**説明:** バックエンドサービス、マイクロサービス、またはサードパーティインテグレーションがプログラムで認証されたが、以下の理由でトークンを使用できなかった。
-認証後、API 呼び出し前のアプリケーション例外
-サービス間のトークン・ハンドオフにおける競合状態
-不適切なダウンストリームエンドポイントによるサービスアカウントの設定ミス

**リスク:** セキュリティの観点からは低いが、信頼性/オブザーバビリティにギャップがある。サービスアカウントが認証されサイレントに失敗すると、セキュリティイベントと運用上の障害の両方が見えなくなる可能性があります。

---

### ランク 5 — 偵察/アカウント列挙 (確率:~ 5%)
**説明:** 攻撃者は、既知または推測された認証情報で認証することにより、有効なアカウントをマッピングしています。認証が成功すると、リソースアクセスアラートを発生させずに、アカウントの存在と認証情報の有効性が確認されます。

**識別指標:**
-短い期間に同じパターンを示す複数のアカウント
-共有インフラストラクチャ (同じ ASN、同じ IP 範囲) からの認証
-その地域やデバイスタイプでは通常認証されないアカウント


-[] **`scanned: 0`の理由を確認 — トークンストア/データベースに直接問い合わせて、該当するタイムウィンドウにトークンが存在するかどうかを確認します。トークンが存在していてもスキャンされなかった場合、検出器は壊れています。
-[] **EventBridge スケジューラの実行ログを確認** — スケジューラによって呼び出された Lambda/Function が正常に実行されたことを確認します。タイムアウトエラー、権限拒否、コールドスタート障害がないか調べてください。
-[] **IAM 権限の検証** — ディテクターの実行ロールがトークンストア (DynamoDB、Redis、RDS — JWT が追跡されるすべての場所) への読み取りアクセス権を持っていることを確認します。
-[] **認証ログを個別に相互参照** — CloudTrail、IdP ログ（Okta、Cognito、Auth0）、またはアプリケーションログに、過去 15 ～ 60 分間の認証イベントを直接照会し、検出器を完全にバイパスします。
