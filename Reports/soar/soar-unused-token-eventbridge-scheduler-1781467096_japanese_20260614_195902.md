### 1a。動作パターン — 認証後の未使用 JWT

| ディメンション | 評価 |
|---|---|
| **重大度** | ミディアム |
| **信頼度** | 低中程度 (JSON あたりのデータ量が不十分) |
| **CVSS プロキシスコア** | ~5.5 (AV: N/AC: H/PR: L/UI: NS: U/C: H/I: L/A: N) |
| **マイター ATT&CK マッピング** | T1078 (有効なアカウント)、T1539 (ウェブセッションクッキーを盗む)、T1550.001 (代替認証マテリアルを使用) |

**正当化:**

認証イベントが JWT を生成するが、しきい値ウィンドウ内でそれ以降の API 呼び出しが 0 回行われるのは、異常な動作シグナルです。通常の使用パターンでは、認証を受けたユーザーは、リソースへのアクセス、アクションの実行、またはセッションの開始という直接的な意図を持ちます。トークンがまったく使用されないということは、いくつかの脅威シナリオのうちの 1 つと考えられます。

-**認証情報の検証**: 攻撃者は、下流のアクティビティベースの検出をトリガーせずに、盗んだ認証情報が有効かどうかをテストしている可能性があります。これは意図的な回避手法です。認証して認証情報の有効性を確認し、後で別のコンテキストやインフラストラクチャから取得したトークンを使用します。
-**オフラインで使用するためのトークン・ハーベスティング**: JWT が発行され、監視対象環境の外で使用するために流出された可能性があります (たとえば、ディテクターのスコープに含まれていないマイクロサービスに対してリプレイされるなど)。
-**認証情報の自動スタッフィングと成功確認**: 認証情報のスタッフィングを行うツールは、認証の成功と悪用を切り離してログに記録することが多く、大規模ではまさにこのパターンが作り出されています。

以下の理由により、行動パターンだけの重大度は「中」に制限されています。
1.正当な説明がある (セクション 2 を参照)
2.悪用は確認されていません。
3.爆発半径は JWT の範囲 (クレーム、オーディエンス、有効期限) に大きく依存します。

ただし、このパターンが**大規模**（複数のアカウント、短い時間枠、類似の IP 範囲または ASN から発信されたもの）に見られると、重大度はすぐに**重要度**にエスカレートします。

### 1b。ディテクタの整合性障害 — スキャンレコードがゼロ

| ディメンション | 評価 |
|---|---|
| **重大度** | 高 |
| **自信** | 高 |
| **影響** | 死角検知 — 範囲不明 |

**理由:**

JSON には「スキャン済み」: 0` と「検索結果の合計」: 0` と表示されます。ゼロレコードをスキャンするディテクターでは、有効な結果は得られません。これは完全な健康診断書ではありません。**これは何も起こらない障害です**。ディテクターは EventBridge スケジューラーによって呼び出され、実行は確認されましたが、データは処理されませんでした。これにより、**検出期間が不明な**という結果になります。つまり、

-この期間中に発生した未使用のトークン・イベントは評価されませんでした
-セキュリティチームが対応範囲を誤って認識している
-（事前の偵察や内部知識などを通じて）ユーザーの検出頻度を把握している攻撃者は、アクティビティのタイミングを計り、スケジューラーのギャップを悪用する可能性があります。

これは、セキュリティカメラが電源をオンにして黒い画面を記録し、「インシデントは発生しませんでした」と報告するのと似ています。

## 2.考えられる説明を可能性別にランク付け

### 行動パターンについて (未使用 JWT)

---

**ランク 1 — 良性:ユーザーがセッションを放棄しました (ほとんどの場合、約 45%) **

ユーザーは認証されたが、アプリケーションを使用する前に中断された。電話がかかってきたり、ブラウザがクラッシュしたり、別の場所に移動したり、単に気が変わったりしただけだった。これは、摩擦の少ない認証 (SSO、生体認証、パスワードレス) を使用する環境 (SSO、生体認証、パスワードレス) で最も統計的に一般的な説明です。この説明の信号対雑音比は、消費者向けのアプリケーションでは高く、API のみの環境や社内ツール環境では低くなります。

*識別指標:*
-このユーザーは1回のみ発生
-認証は営業時間中に行われた
-ユーザーには通常のセッションアクティビティの履歴がある
-地理的異常やデバイス異常なし

---

**ランク 2 — 良性:アプリケーションまたは統合のバグ (~ 25%) **

サービスアカウント、CI/CD パイプライン、または統合レイヤーは正常に認証されたが、ダウンストリームエラー (リダイレクト URI の設定ミス、トークンの解析失敗、最初の API 呼び出し前のネットワークタイムアウト) が原因で認証を続行できなかった。JWT は発行されたが、コンシューマーアプリケーションが受け取ったり処理したりしなかった。

*識別指標:*
-主体（「サブ」）のクレームがサービスアカウントまたは非人身分証明書である
-認証は営業時間外またはバッチウィンドウで行われた
-同じ期間における相関関係のあるアプリケーションエラー
-複数の呼び出しでパターンが繰り返される

---

**ランク 3 — 疑わしい:認証情報検証プローブ (~ 15%) **

有効な可能性のある認証情報 (フィッシング、データ漏えい、またはパスワードのスプレーを通じて入手した) のリストを持っている攻撃者は、リソースアクセス検出をトリガーせずに認証を行い、認証情報の有効性を確認します。JWT は有効な認証情報の「受領」であり、攻撃者はそれを保管したり、盗み出したり、後で使用できるようにアカウントに不正アクセスされたとマークしたりする可能性があります。

*識別指標:*
-通常とは異なるIP、ASN、または位置情報からの認証
-ユーザーエージェント文字列がユーザーの既知のデバイスプロファイルと一致しない
-認証時間がユーザーの行動基準から外れている
-短い時間内に同じパターンを示す複数のアカウント
-このソースからの以前の認証履歴はない

*攻撃経路の精緻化:*

```
攻撃者が認証情報リストを取得 (侵害/フィッシング)
 ↓
自動化ツールはターゲット IdP に対して認証を行います。
 ↓
JWT が発行されました → 攻撃者が HTTP 応答でトークンを取得
 ↓
これ以上リクエストは行われない (アクティビティベースのアラートは発生しない)
 ↓
トークンの保存対象:
 ├── トークン TTL 内でのその後の再生
 ├── オフライン JWT 分析 (アルゴリズムの混乱、弱い秘密のブルートフォース)
 └── 認証情報の確認 → 標的型攻撃へのエスカレーション
```

---

**ランク 4 — 疑わしい:外部リプレイのためのトークンの流出 (~ 10%) **

JWT は発行され、すぐに外部システムに流出しました。攻撃者は、監視対象環境の外で、つまり検出器がカバーしていないマイクロサービス、内部 API、またはリソースに対してトークンを使用しようとしています。これは、JWT オーディエンス (「aud」) のクレームが幅広く存在する環境や、同じ JWT が複数のサービスで受け入れられている環境では特に危険です。

*ブラスト半径の考慮事項:*
JWT の範囲が広い場合や、複数のサービスで受け入れられている場合 (署名鍵を共有するマイクロサービスアーキテクチャでは一般的)、1 つの漏洩トークンで多数のシステムへのアクセスが許可される可能性があります。爆発半径は次の値で直接スケーリングされます。
-トークン TTL (長い = 露出ウィンドウが長い)
-スコープの幅 (管理者と読み取り専用)
-トークンを受け入れるサービスの数
-トークン取り消しインフラストラクチャが存在するかどうか

---
