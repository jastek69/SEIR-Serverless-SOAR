## イベントコンテキストに関する予備注記

詳細な分析に進む前に、重要な点を明らかにしておく必要があります。**説明で説明されているセキュリティイベントは、提供された JSON コンテキストと直接矛盾します。** 説明には、トークンが発行され、15 分以内に未使用になったことが記載されていますが、JSON テレメトリーでは次のことがわかります。

-`scanned: 0` — トークンは評価されませんでした
-`alerted: 0` — アラートは生成されませんでした
-`結果_total: 0` — 結果はありません
-`結果サンプル:[] `— 空の結果セット
-`閾値_分:5` — 検出器の閾値は 15 分ではなく 5 分です

この不一致自体が、セキュリティ関連の所見です。以下の分析では、説明で説明されている理論上のシナリオと、JSON テレメトリに存在する運用上の異常の**両方**を対象としています。どちらも調査に値するからです。

### JSON テレメトリアノマリー (検出器でスキャンがゼロ)
| 寸法 | 評価 | ジャスティフィケーション |
|---|---|---|

## 2。考えられる説明を可能性別にランク付け

### ナラティブイベント用 (トークンは発行済みだが未使用)

#### 🥇 ランク 1 — 良性:ユーザーがセッションを放棄しました (ほとんどの場合、約 45%)
ユーザーは認証されたのに別の場所に移動したり、ブラウザタブを閉じたり、接続が切断されたり、意図したアクションを完了する前に中断されたりしました。これは、エンタープライズ環境、特にログインフローが複雑な Web アプリケーションで最も統計的に一般的な説明です。

-**サポート指標:** 認証イベントは 1 回、それ以降の API 呼び出しなし、地理的な異常なし
-**リスクレベル:** ごくわずか
-**検出値:** 単独では信号対雑音比が低い

#### 🥈 ランク2 — 良性:自動ヘルスチェックまたは統合テスト (~ 20%)
CI/CD パイプライン、合成監視ツール、または統合テストスイートでは、機能的な操作にトークンを使用せずに、認証情報の有効性を検証することがよくあります。

-**サポート指標:** サービスアカウントからの認証、人間以外のユーザーエージェント、スケジュールされたタイミング
-**リスクレベル:** 低い。ただし、テストで実際の認証情報を使用する場合、認証情報の衛生上の問題が発生する可能性がある
-**検出値:** ベースラインを作成し、適切なタグを付けて抑制する必要がある

#### 🥉 ランク 3 — 疑わしい:クレデンシャルスタッフィング/有効性プローブ (~ 15%)
攻撃者が (侵害、フィッシング、またはダークウェブ市場での購入を通じて) 認証情報を入手した場合、より標的を絞った攻撃に利用する前に、認証情報が有効であることを確認するだけで認証を受ける可能性があります。攻撃者のツールは有効なアカウントのリストを作成中という偵察段階にあるため、トークンは使用されません。
-**サポート指標：** 通常とは異なるIP/ASNからの認証、営業時間外のタイミング、アカウント間の複数の類似イベント、ユーザーエージェントの異常
-**攻撃経路:**
 ```
 認証情報リストの取得 → 自動認証プローブ → 
 有効な認証情報の確認 → 標的型悪用 
 (データ漏洩、権限昇格、ラテラルムーブメント)
 ```
-**爆発半径:** これが大規模な認証情報の盗用だと、多数のアカウントが同時に侵害される可能性があります
-**リスクレベル:** 確認されれば高い

#### ランク 4 — 疑わしい:繰延使用のためのトークン・ハーベスティング (~ 10%)
認証フローにアクセスできる攻撃者が（侵害されたクライアントデバイス、Man-in-the-Browserマルウェア、不正なアプリケーションなどを介して）、発行されたJWTをキャプチャして、監視対象環境の外で使用する可能性があります。トークンは「未使用」のように見えます。これは、消費が監視されていないチャネルで行われているためです。

-**サポート指標:** 以前に異常な動作を示したクライアントに発行されたトークン、有効期限が長いトークン、その後の呼び出しが予定されていないトークン
-**攻撃経路:**
 ```
 マルウェア/不正アプリによる認証フローの傍受→ 
 JWT が攻撃者のインフラストラクチャに流出 → 
 攻撃者が制御するホストの API に対して使用されるトークン → 
 元のセッションコンテキストでは「未使用」と表示される
 ```
-**リスクレベル:** 確認されれば高 — 検出ギャップのある活発な悪用を示す

#### ランク 5 — 運用:アプリケーションのバグまたは設定ミス (~ 7%)
クライアントアプリケーションのバグが原因で、認証は成功しても、後続のリクエストにはトークンを添付できないことがあります (例:スコープ外の変数に格納されたトークン、不適切なヘッダー構造、非同期コードの競合状態など)。

-**リスクレベル:** セキュリティの観点からは低いが、信頼性に問題がある

#### ランク 6 — 疑わしい:内部脅威の偵察 (~ 3%)
正規のユーザーは、通常の操作を行わずに、アクセスレベルを確認したり、アカウントにフラグが付けられているかどうかをテストしたりするために認証を行います。

-**リスクレベル:** 中程度 — 行動基準の比較が必要

### JSON テレメトリアノマリ (スキャン済み:0) の場合

#### 🥇 ランク 1 — 運用中:トークン・ストアが空/アクティブなセッションなし (~ 40%)
検出器は実行されましたが、データストアに評価対象のトークンが見つかりませんでした。スキャン時にシステムにアクティブなセッションがない場合は問題ありませんが、認証ログと照合する必要があります。

#### 🥈 ランク 2 — 設定エラー:クエリまたはデータソースポインタが正しくありません (~ 35%)
ディテクタは、間違ったデータベーステーブル、インデックス、間違った時間枠、または誤って設定されたデータソースをクエリしています。これはサイレントエラーです。ディテクタは成功を報告しますが、意味のある評価はしていません。

#### 🥉 ランク 3 — セキュリティ上の懸念:検出器の抑制または改ざん (~ 15%)
十分なアクセス権を持つ攻撃者が、ディテクターの設定を変更したり、トークンストアをクリアしたり、EventBridge ルールを操作してアクティビティが検出されないようにしたりした可能性があります。

#### ランク 4 — インフラストラクチャの問題:データパイプラインの障害 (~ 10%)
パイプラインの破損、Lambda 関数の障害、またはイベントルーティングの設定ミスが原因で、トークン発行イベントがディテクターがクエリしたデータストアに届きません。
