## イベントコンテキストに関する予備注記

詳細な分析に進む前に、**提供された JSON コンテキストから、スキャンされたレコード、アラート、そして何も検出されない検出器の呼び出しが明らかになるという重要な事実を明らかにしておく必要があります。** これが分析を大きく左右します。説明イベントの説明（トークンが発行され、15 分以内に使用されなかった）は、このシステムが検出するように設計されている「意図された検出シナリオ」を表していますが、現在の実行ではデータが生成されませんでした。検出器が対処する脅威モデルと検出器自体の動作状態という両方の側面を分析する必要があります。

## 1。重要度評価

### 主な脅威シナリオ（検出器が検知するように設計されている対象）

| ディメンション | レーティング | ジャスティフィケーション |
|---|---|---|
| **固有の脅威の重大度** | **中 (3/5) ** | 未使用のトークンだけでは弱いシグナルであり、状況によってエスカレーションが決まる |
| **検出ギャップリスク** | **高 (4/5) ** | 検出器が誤動作すると、攻撃クラス全体が監視対象外になる |
| **オペレーショナルリスク (現在の稼働状況) ** | **高 (4/5) ** | スキャンされたレコードがゼロの場合は異常で、パイプラインに障害が発生している可能性がある |
| **複合リスク** | **高** | 検出器のヘルス障害は全体的なリスク状態を高める |

### 正当化

JWT を生成するけれども JWT を実行しない認証済みセッションは、**低シグナル、ハイコンテクスト** のセキュリティイベントが交わる興味深い場所にあります。単独では問題ありません。ユーザーがセッションを放棄したり、ブラウザーを閉じたり、常にネットワークの中断を経験したりします。ただし、規模が大きくなったり、他の指標と組み合わせたりすると、次のような意味のあるシグナルになります。

-**クレデンシャルスタッフィング検証実行** — 攻撃者は、ダウンストリームのアクティビティベースの検出を発生させずに、認証情報を認証してクレデンシャルの有効性を確認します
-**トークン・ハーベスティング** — 使用前にトークンが漏洩する、侵害された認証フロー
-**自動偵察** — ボットが認証エンドポイントを調べて有効なアカウントをマッピングする

**threshold_minutes: 5** (15 分の説明よりも厳しい) によると、エンジニアリングチームはすでにこの検出器を、認証情報の乱用パターンに適した短期間の不正使用を検知するように調整済みであることがわかります。

**scanned: 0**という結果が、より緊急の懸案事項です。レコードを処理しないスケジュールされたディテクタは、評価するトークンがまったくないか (トラフィックの少ない環境ではもっともらしい)、ディテクタに供給されるデータパイプラインが壊れているかのどちらかです。つまり、実際に悪質な未使用のトークンが見逃されてしまいます。

## 2。考えられる説明を可能性別にランク付け

### ゼロスキャン検出器の状態について

| ランク | 説明 | 発生可能性 | 証拠の根拠 |
|---|---|---|---|
| 1 | **評価ウィンドウにトークンが存在しません** | 高 | トラフィックの少ない環境、営業時間外のスケジューラ実行、またはスキャンの前にすべてのトークンが使用された/期限切れになった |
| 2 | **設定ミスのためデータソースクエリが空を返しました** | Medium-High | タイムウィンドウフィルタが正しくない、テーブル/インデックスが間違っている、または IAM 権限のリグレッションが発生している |
| 3 | **トークンストアが利用できないか、アクセスできない** | ミディアム | データベース接続障害、Redis キャッシュダウン、またはネットワークパーティション |
| 4 | **トークンの取り込みパイプラインが完了する前にスケジューラーが起動されました** | Medium | トークンの書き込みと検出器の読み取りの競合状態 |
| 5 | **最近のデプロイで発生したディテクターコードのバグ** | Low-Medium | スキャンクエリのロジックエラーで何も表示されずに空のセットが返される |
| 6 | **内部関係者または攻撃者による意図的な抑制** | 低 | アクセス権限の昇格が必要。他に異常がある場合は除外する価値がある |

### 脅威シナリオの場合 (トークンは発行されたが、使用されたことはない)
| ランク | 説明 | 可能性 | エビデンスの根拠 |
|---|---|---|---|
| 1 | **良性のユーザー放棄** | 非常に高い | ユーザー認証済み、中断された、タブを閉じた、または接続が失われた |
| 2 | **自動ヘルスチェック/合成モニタリング** | High | CI/CD パイプライン、アップタイムモニター、およびロードバランサーは、トークンを消費せずに認証を行うことが多い |
| 3 | **クレデンシャル・スタッフィング — 有効性チェックのみ** | Medium | 攻撃者は認証情報が正常に動作していることを確認する。目的が列挙であれば、トークンを使用する必要はない |
| 4 | **使用前のトークンの流出** | Low-Medium | MITM または侵害されたクライアントがトークンを傍受し、攻撃者が別のインフラストラクチャからトークンを利用する |
| 5 | **開発者による認証フローテスト** | Low-Medium | 開発者がワークフローを完了せずに本番環境でログインフローをテストする |
| 6 | **壊れたクライアントアプリケーション** | 低 | クライアントがトークンを受け取っても正しく保存または送信できない |
| 7 | **アカウント乗っ取り調査** | Low | 攻撃者が大規模なキャンペーンを開始する前に盗んだ認証情報をテストする |

## 3。アナリストが推奨するアクション

### 即時 (0 ～ 1 時間)

#### ステップ 1: ディテクタの正常性を検証する
これは `scanned: 0`の場合で最も優先度の高いアクションです。

```
□ ディテクターを実行している Lambda/サービスからトークンストア (DB/キャッシュ) にアクセスできることを確認する
□ ディテクターの基になるクエリをトークンストアに対して手動で実行する
□ CloudWatch Logs でディテクターの Lambda 実行を確認 — 例外、タイムアウト、空のクエリ結果がないか調べる
□ IAM ロールの権限が変更されていないことを確認する (トークンテーブルの GetItem、Query、Scan)
□ EventBridge スケジューラーが正しい時間に起動され、呼び出しが制限されていないことを確認する
□ 最近のデプロイでトークンスキーマ、テーブル名、またはインデックス構造が変更されていないか確認する
```

#### ステップ 2: 認証ログのクロスリファレンス
ディテクタが正常で「scanned: 0」が正当であれば:

```
□ IdP/Auth サービスのログを照会して、過去 60 分間に成功した認証をすべて調べる
□ 発行されたが対応するダウンストリーム API 呼び出しがないトークンを特定する
□ ソース IP を脅威インテリジェンスフィードと関連付ける
□ 認証速度の異常をチェック (同じ IP で複数のアカウントを認証)
```

#### ステップ 3: ベースラインの確立
```
□ 環境内の「未使用トークン」イベントの標準発生率を確認
□ これが新しい検出器である場合は、しきい値を調整する前に1日目のベースラインを設定してください。
□ 予想されるゼロスキャンシナリオ（メンテナンスウィンドウ、営業時間外など）を文書化する
```

### 短期 (1 ～ 24 時間)

```
□ スキャン時に明示的なアラートを追加:0 は既知のメンテナンス時間外に発生
□ 検出器の有効性を検証するために、カナリアトークン（スキャン時に常に表示されるべき合成トークン）を実装する
□ 過去 7 日間の認証ログを確認して、未使用のトークンのパターンがないかどうかを確認します。
□ 認証情報の乱用が疑われる場合は、認証エンドポイントの悪用がないかWAFログと相互参照する
□ トークンの失効が機能していることを確認する — 未使用のトークンはthreshold d_minutesが経過したら無効にする必要がある
```
