# Job Failure Post-Mortem

## Summary

A `comfyui_gen` job (ID: `22025822-f804-4b72-9cd6-b8762ae75fc8`) submitted by `admin.test` stalled after the GPU worker's heartbeat went silent. The job recorded 10.1 seconds of execution and uploaded 1 output before the worker stopped responding, leaving the job in a `stalled` state. The failure was detected at approximately 2025-05-14 ~16:58 UTC based on the `updated_at` timestamp.

## Probable Cause

**Confidence: Low — evidence is thin.**

The worker completed at least one inference step (1 output uploaded, 10.1 s duration recorded) before going silent. This is consistent with a **post-inference worker crash or OOM event** on the GPU instance — e.g., the process was killed after the generation completed but before it could write a final `completed` status back to DynamoDB.

However, the control-plane logs contain only the initial submission event. **No worker-side logs are available in this context** (they reside in `journald` on the GPU instance, not CloudWatch). The following evidence is missing and is required to confirm the cause:

- `journalctl -u comfyui-worker` output from the GPU instance around `1784198338`
- GPU instance system logs (OOM killer activity, CUDA errors, instance health events)
- Whether the job's output in `outputs/jobs/22025822-f804-4b72-9cd6-b8762ae75fc8/` is complete or partial
- Any EC2/instance-level events (spot interruption, hardware fault, auto-scaling termination)

---

## Recommended Actions

1. **Retrieve worker logs immediately** — SSH to the GPU instance that processed this job and run `journalctl -u comfyui-worker --since "2025-05-14 16:57:00" --until "2025-05-14 17:05:00"`. This is the single highest-value action and will likely reveal the exact failure mode (crash, OOM, CUDA error, etc.).

2. **Check for OOM or spot interruption on the GPU instance** — Run `journalctl -k | grep -i "oom\|killed\|spot\|terminated"` and review EC2 instance health history in the AWS console for the relevant instance around `16:58 UTC`.

3. **Verify output completeness** — Inspect the S3 prefix `outputs/jobs/22025822-f804-4b72-9cd6-b8762ae75fc8/` to determine whether the 1 uploaded output is a valid final artifact or a partial/intermediate file. This determines whether the job needs to be retried.

4. **Implement worker heartbeat forwarding to CloudWatch** — The current architecture leaves worker runtime logs only in `journald`, making post-mortems dependent on instance availability. Add a CloudWatch Logs agent or structured heartbeat metric emission from the worker to enable faster, instance-independent diagnosis.

5. **Add a post-upload status-commit step with retry logic** — The pattern of "output uploaded but status never set to `completed`" suggests the worker may lack a reliable final commit step. Ensure the worker retries the DynamoDB status update independently of the inference process, so a crash after upload does not leave jobs permanently stalled.

6. **Retry the job if output is incomplete** — If step 3 confirms the output is unusable, re-enqueue the job for `admin.test`. If the instance is confirmed unhealthy, quarantine it before routing new work.
