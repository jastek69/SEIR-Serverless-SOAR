## イベントコンテキストに関する予備注記

詳細な分析に進む前に、重要な点を明らかにしておく必要があります。**説明で説明されているセキュリティイベント（ユーザー認証済み、JWT発行済み、15分以内にトークン未使用）は、提供されたJSONコンテキストと一致しません。** JSONには、`scanned: 0`、`alerted: 0`、`findings_total: 0`、空の `findings_sample` と表示されます。つまり、検出器は実行されたが、**分析対象が何もありませんでした**つまり、トークンはまったくスキャンされませんでした。

**理由:**

## 2。考えられる説明を可能性別にランク付け

### 実際のイベントについて (スキャン回数ゼロ、検出結果ゼロ)

#### ランク 1 — トークンストアクエリの失敗または設定ミス * (約 45% の可能性) *
ディテクターはイベントブリッジスケジューラー経由で正常に呼び出されましたが、トークンストア (Redis、DynamoDB、RDS など) への接続またはトークンストアへのクエリが失敗しました。これには以下の原因が考えられます。
-ディテクターを実行している Lambda/Compute の IAM ロール権限の期限切れまたは取り消し
-VPC の設定ミスによりデータベースへのアクセスが妨げられています
-トークンストアスキーマの変更によりクエリが中断される
-環境変数の設定ミス (テーブル名、エンドポイント、またはインデックスが間違っている)

**これを裏付ける証拠:** JSON にエラーが何も表示されずに「scanned: 0」ということは、スキャンループに入らなかったか、すぐに終了したかのどちらかであることを示しています。実行時例外 (通常は異なるイベント構造を生成する) ではなく、壊れたクエリから返された空の結果セットと一致します。

#### ランク 2 — トークン・ライフサイクル・アーキテクチャー・ギャップ* (約 30% の確率) *
システムは、発行時に JWT メタデータをクエリ可能なストアに永続化していない可能性があります。JWT がステートレス (トークン・レジストリのない標準 RS256/HS256) の場合、スキャンするレコードはありません。ディテクターは、トークン・レジストリーは存在するが、レジストリーが空か、何も入力されていないか、またはディテクターが間違ったデータソースを指していることを前提に構築されました。

**リスクへの影響:** ステートレス JWT は設計上、トークン・レジストリーがないと取り消したり追跡したりできません。組織が、このディテクターがカバレッジを提供していると考えているのに、レジストリーに何も入力されていない場合、検出機能全体が幻想です。
#### ランク 3 — スケジューラの設定ミス/不適切な環境* (~ 15% の可能性) *
EventBridge Scheduler が、非本番環境、廃止予定のエンドポイント、または認証アクティビティが行われないリージョンに対してディテクターをトリガーしている可能性があります。ディテクターは空のデータセットに対して正常に動作し、成功を報告します。

#### ランク 4 — オフピーク時における意図的な行動* (~ 10% の確率) *
スケジューラーがメンテナンス時間中または認証アクティビティがまったくない期間に起動した場合、「scanned: 0」が正当である可能性があります。ただし、これは同じ時間帯の認証ログと照合して検証する必要があります。

## 3。アナリストが推奨するアクション

### 即時アクション (0 ～ 2 時間)

```
優先度1: 検出器の動作状態を確認
```

1。**ディテクター実行ログを検査** —「unused_token_detector」を実行している Lambda/サービスの CloudWatch ログ (または同等のもの) を取得します。以下を探してください。
 -イベント JSON に例外やスタックトレースが表示されない
 -データベース接続タイムアウトまたはパーミッション拒否エラー
 -結果セットが空か、クエリが実行されなかったか

2.**トークンのレジストリ登録を確認** — トークンストアに直接問い合わせる:
 ```sql
 --例:トークンレジストリにレコードがあるかどうかを確認する
 カウント (*)、最大値 (発行時)、最小値 (発行時) を選択します
 jwt_token_registry から
 発行場所 >= NOW ()-間隔 '24 時間';
 ```
 これによって返される行が 0 行でも認証ログに成功したと表示される場合、レジストリーにはデータが入力されていません。**これは P1 の結果です**。

3.**クロスリファレンス認証ログ** — 検出器がスキャンしたのと同じタイムウィンドウの認証イベントを取得します。
 ```
 タイムウィンドウ:[スケジューラ_トリガー時間-しきい値_分] から [スケジューラ_トリガー時間]
 クエリ:event_type = 'TOKEN_ISSUED' でタイムスタンプがウィンドウ内にある認証イベント
 ```
 認証イベントが存在するが `scanned: 0` の場合、ディテクタはデータパイプラインが中断されます。

4.**IAM 権限の確認** — ディテクターの実行ロールに、トークンストアからの読み取りに関する現在の有効な権限があることを確認します。最近のポリシー変更を確認してください。

### 短期的なアクション (2～24 時間)

5.**EventBridge スケジューラー設定の確認** — スケジューラーが正しい環境、地域、機能 ARN をターゲットにしていることを確認します。スケジュール条件式が意図した頻度で実行されていることを確認します。

6.**検出器セルフヘルスレポートの実装** — 認証アクティビティが存在することがわかっている時間帯に「scanned: 0」が発生すると、検出器はメトリクスまたはアラームを発する必要があります。何も検出せずに何もスキャンしない検出器は、静かな環境で動作する検出器と区別がつきません。

7.**SIEM との相関関係** — 仮定の未使用トークンのシナリオが当てはまる場合は、ユーザー ID に注目してください。
 -異なる IP からの同時セッションの有無を確認
 -以前の認証履歴に異常がないか確認する
 -同じウィンドウでパスワードリセットイベントや MFA 変更イベントを確認

### 進行中のアクション

8.**ベースラインメトリクスの確立** — 認証量に基づいて、時間枠ごとに予想される「スキャン済み」回数を定義します。偏差が 2 標準偏差を超えると警告します。
