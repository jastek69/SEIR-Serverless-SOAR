## イベントコンテキストに関する予備注記

詳細な分析に進む前に、重要な点を明らかにしておく必要があります。**説明で説明されているセキュリティイベント（ユーザー認証済み、JWT発行済み、15分以内にトークン未使用）は、提供された JSON コンテキストと一致しません。** JSON には、`scanned: 0`、`alerted: 0`、`findings_total: 0`、`findings_sample: [] `が表示されます。つまり、検出器は実行されたが、**分析対象が何もありませんでした**つまり、トークンはまったくスキャンされませんでした。

この不一致自体はセキュリティ関連の所見であり、この分析全体を通して主要な懸念事項として扱われます。この分析では、説明で説明されている仮想イベント**と、JSON で示された検出器の誤動作**の両方に対処します。

## 1。重要度評価

### ナラティブイベント (未使用 JWT トークン)
| 属性 | 値 |
|---|---|
| **重大度** | 低—中 (エスカレーションの可能性がある情報提供) |
| **信頼度** | 低 (悪意のある意図を確認するにはデータが不十分) |
| **緊急性** | LOW (アクティブな悪用は確認されていません) |

**正当化:**
認証されたセッションで JWT が発行されたのに一度も使用されていないということは、単独では弱いシグナルです。それが意味を持つのは、総合的にみた場合か、他の行動指標と相関する場合のみです。1 回発生しただけで、無害な説明 (ユーザーがブラウザを閉じた、ネットワークドロップ、ボットスキャンなど) を示す基準率が高くなります。ただし、大規模な場合や、クレデンシャル・スタッフィング・インジケータとの組み合わせでは、偵察パターンやトークン・ハーベスティング・パターンを表す場合があります。

### JSON コンテキスト検索 (ディテクタの誤動作)
| 属性 | 値 |
|---|---|
| **重大度** | 高 |
| **自信** | 高 |
| **緊急** | 高 |

**正当化:**
検出器が起動してもレコードがまったくスキャンされない場合、**セキュリティコントロールのサイレントエラー**と見なされます。これは、本来検出するように設計されていたイベントよりも明らかに危険です。もし「scanned: 0」が一時的な空白ウィンドウではなく永続的な状態を反映している場合、その組織には未知の期間で検出ギャップが生じていることになります。このギャップを発見した攻撃者は、アラートを発生させることなく自由にトークンを発行して使用できます。壊れたディテクターの爆発範囲は、JWT 認証を受けたアタックサーフェス全体です。

## 2。考えられる説明を可能性別にランク付け

### ナラティブイベント用 (未使用の JWT トークン)

#### ランク 1 — 良性のユーザー行動 (確率:~ 60%)
統計的に最も可能性の高い説明です。ユーザーが認証され、次の操作が行われました。
-タスクを完了する前にブラウザタブを閉じた
-中断 (電話、会議、システムクラッシュ)
-スリープ状態になった、または接続が切れたモバイルデバイスから認証された
-クライアント側のレンダリングエラーにより、トークンを後続のリクエストに添付できなくなった

**検出ギャップ:** これは個々のイベントレベルでの悪意のある動作と区別がつきません。信号とノイズを分離するには、音量と速度の解析が必要です。

#### ランク 2 — 自動認証情報検証/資格スタッフィング (確率:~ 20%)
クレデンシャルスタッフィングキャンペーンを実施している攻撃者は、セッションをすぐに使用せずに認証して認証情報を検証することがよくあります。ワークフローは以下のとおりです。
1.認証情報ペアを認証エンドポイントに送信する
2.JWT を受け取る (有効な認証情報の確認)
3.認証情報のペアを「有効な」リストに保存して、後で利用したり販売したりできるようにする
4.次の認証情報ペアに移動

攻撃者の目的は**セッションの使用ではなく検証**であったため、トークンは決して使用されません。このパターンは、以下の理由で特に危険です。
-認証後の動作検出をトリガーせずにアカウント侵害を確認できる
-有効な認証情報リストは、数日後または数週間後に別のインフラストラクチャから使用される可能性があります
-スタッフィングが遅く、分散していると、認証エンドポイントのレート制限がキャッチされない場合がある
**相関する指標:**
-ソース IP レピュテーション (住宅用プロキシ、Tor 出口ノード、既知のボットネット)
-ユーザーエージェント文字列の異常
-地理的不可能性または速度
-短期間に同じ IP から複数のアカウントが認証された
-最近のログイン履歴がないアカウントに対する認証試行

#### ランク 3 — トークン・ハーベスティング/セッション固定の準備 (確率:~ 10%)
特に以下の場合、より高度な攻撃者が JWT を入手して後で使用できるように認証を受ける可能性があります。
-JWT の有効期限が長い (24 時間以上など)
-攻撃者が特定の時間帯 (営業時間外など) までトークンを使用するのを待っている
-攻撃者は、トークンがログアウトイベント後も存続するかどうかをテストしている (トークンの無効化が不適切でないかを確認している)

これは確率は低いが影響の大きいシナリオです。JWT の存続期間が長く、トークン・ストアがサーバー側の失効をサポートしていない場合、攻撃者は、リセットによって既存のトークンが無効にならない限り、パスワードのリセット後も存続する永続的な認証情報を持っています。

#### ランク 4 — 自動セキュリティテスト/侵入テスト (確率:~ 7%)
内部または承認された外部のセキュリティテストによってこのパターンが発生する可能性があります。ユーザーフローをすべて完了せずに認証エンドポイントをテストするスキャナーは、未使用のトークンを生成します。これは既知のテストスケジュールに照らして検証できるはずです。

#### ランク 5 — アプリケーションバグ/クライアント側の障害 (確率:~ 3%)
クライアントアプリケーションのバグにより、トークンは発行されても正しく保存されない可能性があります (「LocalStorage」の書き込み失敗、SameSite ポリシーにより Cookie が設定されない、CORS の問題によりクライアントにトークンを返却できないなど)。サーバーがトークンを発行したにもかかわらず、ユーザーには認証が失敗します。

### JSON コンテキスト検索 (ディテクタの誤動作) 用

#### ランク 1 — データソースの切断 (確率:~ 45%)
ディテクターはトークンストアまたは認証イベントログに接続されていません。考えられる原因:
-IAM 権限の変更により、トークンデータベースまたは DynamoDB テーブルへのディテクターの読み取りアクセスが取り消された
-トークンのストレージメカニズムが変更され (Redis から DynamoDB への移行など)、ディテクターは更新されませんでした
-EventBridge スケジューラーは正しい Lambda ARN を呼び出しているが、データソースを指している Lambda の環境変数が正しく設定されていないか、古くなっている

#### ランク 2 — 空き時間枠/競合状態 (確率:~ 25%)
スケジューラは、監視対象状態のトークンが存在しない時間帯に起動しました (たとえば、トークン TTL クリーンアップが検出される前に実行されたり、システムのトラフィック量が少ない場合など)。これは問題ありませんが、過去のスキャン回数と照らし合わせて検証する必要があります。

#### ランク 3 — Lambda 実行エラーが飲み込まれた (確率:~ 20%)
Lambda 関数はスキャンフェーズ中にエラーに遭遇し、デフォルトまたは空の結果オブジェクトを返しましたが、エラーはアラートレイヤーに伝わりませんでした。これは例外処理の範囲が広すぎるコードレベルの障害です。

#### ランク 4 — 意図的なスコープの縮小が伝えられていない (確率:~ 10%)
開発者がセキュリティチームに変更を伝えずにクエリスコープを変更した (たとえば、現在のトークンをすべて除外するフィルターを追加するなど)。ディテクターは正常に動作しますが、結果セットが空になるように設計されているため、セキュリティへの影響は確認されていません。
