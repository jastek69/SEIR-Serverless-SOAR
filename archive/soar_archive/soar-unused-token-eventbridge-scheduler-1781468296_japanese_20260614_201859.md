## イベントコンテキストに関する予備注記

完全な分析に進む前に、重要な観察結果を明らかにする必要があります。**JSON コンテキストから、スキャンされたトークンがゼロ、アラートがゼロ、検出結果がゼロのディテクター実行が明らかになります。** これ自体が重要なシグナルであり、必ずしも攻撃を示すものではありませんが、**監視パイプライン自体に検出ギャップや設定ミスがある可能性があります。** 分析では、理論上のイベントパターン (認証後に未使用の JWT) と運用上の異常の両方に対処します。検出器出力で。

## 1。重要度評価

### 全体的な重要度:**中** (調査待ちで「高」にエスカレーションする可能性あり)

| ディメンション | 評価 | 正当化 |
|---|---|---|
| 機密保持への影響 | 中程度 | トークンは使用されずに流出された可能性がある |
| 完全性への影響 | 低中程度 | 誤用は確認されていないが、トークンは有効なままである |
| 可用性への影響 | 低 | サービスの中断は見られなかった |
| 検出信頼度 | **低い** | 検出器が 0 トークンをスキャンしました — カバレッジギャップが確認されました |
| 爆発半径 | 中 | トークンの範囲/クレームにより異なる |

### 正当化

未使用トークンのパターンは、**良性のユーザー行動**と**悪用前の偵察**の興味深い交点にあります。未使用のトークンが 1 つあること自体は、低いシグナルです。ただし、以下の組み合わせは

1.認証成功イベント (認証情報の有効性が確認された)
2.トークンの発行 (アタックサーフェスの作成)
3.それ以降の API 呼び出しはゼロ (正当な使用では異常)
4.**何もスキャンしなかった検出器** (死角の監視)

... 受動的モニタリングではなく積極的な調査が必要な複合リスクプロファイルを作成する。

イベント説明で言及されている**15分のしきい値**とJSONの**5分のしきい値** (`threshold_minutes: 5`) は、それ自体が報告に値する違いです。アラートロジックの構成ミスやドキュメントドリフトを示している可能性があります。

## 2。考えられる説明を可能性別にランク付け

### ランク 1 — 良性:ユーザーがセッションを放棄した回数 (確率:~ 45%)

**シナリオ:** 正当なユーザーが認証され (ログインリンクをクリックしたり、SSO を開始したりするなど)、ブラウザタブを閉じたり、接続が切断されたり、目的のワークフローを完了する前に中断されたりしました。

**サポート指標:**
-アカウントに以前に異常なアクティビティがあったことはない
-既知の IP/デバイスフィンガープリントによる認証
-単独で発生した事象

**依然として重要な理由:** 無害な未使用のトークンであっても、攻撃対象期間を延長する**ライブ認証情報**です。何時間も有効であっても一度も使用されていないトークンは、負債となります。

---

### ランク 2 — 自動/ボット認証プローブ (確率:~ 25%)

**シナリオ:** 自動スクリプトまたは認証情報収集ボットが、有効な認証情報 (以前の侵害、フィッシング、またはダークウェブ市場での購入によって入手したもの) を使用して正常に認証されました。ボットは認証情報が正しく機能することを検証しましたが、悪用段階には進みませんでした。意図的に (認証情報検証サービス)、あるいは使用前にレート制限/ブロックされていたためです。

**攻撃経路:**
```
クレデンシャル・スタッフィング・ボット
 → POST /認証/ログイン [有効な認証情報]
 → JWT を受信します
 → 攻撃者のインフラに「有効な認証情報」を記録する
 → トークンを使用しない (行動分析を発生させない)
 → トークンは未使用で期限切れになる
 → 攻撃者は後で新しい認証を行って戻ってくる
```

**これが危険な理由:** このパターンは、認証後の異常を探す**動作検出**を回避するために特別に設計されています。攻撃者は API 不正使用検出器をトリガーせずに認証情報の有効性を確認します。

---

### ランク 3 — トークンの流出 (使用前) (確率:~ 15%)

**シナリオ:** トークンは (XSS、マンインザブラウザ、侵害されたクライアントエンドポイント、またはネットワーク傍受によって) 発行および流出したが、攻撃者はまだ使用していない。15 分という時間枠は単に監視期間に過ぎず、検出期間の終了後にトークンが使用される可能性もあります。
**攻撃経路:**
```
正規ユーザー認証
 → JWT が発行され、ブラウザに保存される (ローカルストレージ/クッキー)
 → 悪意のあるスクリプト (XSS) がトークンを攻撃者 C2 に流出させる
 → ユーザがセッションを終了するか、トークンを正当な側で未使用の状態にする
 → 攻撃者は検出期間終了後に別の IP/UA からのトークンを使用する
```

**検出ギャップ:** 15 分間の未使用トークンのチェック期間終了後にトークンが使用された場合、このイベントは再トリガーされません。攻撃者は実質的にトークンの有効期限が切れるまで待つ必要があります。

---

### ランク 4 — 検出器の設定ミス/偽陰性 (確率:~ 10%)

**シナリオ:** JSON の `scanned: 0` フィールドは、検出器自体が壊れている可能性があることを強く示唆しています。これは次のことを意味する可能性があります。

-EventBridge スケジューラーは正常に起動したが、Lambda/Worker にクエリするデータソースがない（トークンストアが空である、DynamoDB テーブルが間違っている、Redis キースペースが正しくない）
-ディテクターが照会するトークンレジストリーにトークン発行が書き込まれていない
-デプロイメントまたは構成の変更により、トークントラッキングパイプラインが壊れました。

**これは、分析をトリガーした特定のイベントに関係なく、未使用トークンの検出機能がまったく機能しない可能性があることを意味するため、運用上最も緊急な調査結果です**。

---

### ランク 5 — 内部脅威/権限昇格ステージング (確率:~ 5%)

**シナリオ:** 内部アカウントまたは侵害された特権アカウントは、昇格したクレームを含むトークンを取得するために認証を行い、実行予定の特定のアクションとの相関を避けるために意図的に使用を遅らせます。

## 3。アナリストが推奨するアクション

### 即時アクション (0 ～ 1 時間)

#### ステップ 1: まず検出器の異常を調査する

「scanned: 0」という結果が、運用上の最優先課題です。

```
チェックリスト:
[] トークン・レジストリーのデータストアにディテクター Lambda からアクセスできることを確認します。
[] トークン発行イベントがレジストリに書き込まれていることを確認する
[] クラウドウォッチのログをチェックして、unused_token_detectorラムダが実行されているかどうかを確認します
[] イベントブリッジスケジューラーのターゲット ARN と IAM の権限を確認する
[] ディテクターのクエリロジックがトークンストアスキーマと一致することを確認する
[] 相互参照：認証ログごとに過去24時間に発行されたトークンはありますか？
```

#### ステップ 2: 特定の認証イベントを特定する

```
クエリ認証ログ:
-抽出:ユーザーID、ソースIP、ユーザーエージェント、ジオロケーション、デバイスフィンガープリント
-タイムウィンドウ:認証タイムスタンプ ± 30 分
-チェック:この IP/UA の組み合わせは、以前に成功した認証でも見られましたか?
-チェック:MFA が使用されたか、バイパスされたか?
-チェック:トークンの有効期限、適用範囲、クレームは?
```

#### ステップ 3: トークンリスクの評価

```
トークン・トリアージ:
[] トークンはまだ有効ですか？→「はい」の場合は、即時取り消しを検討してください。
[] トークンにはどのような範囲/役割がありますか？
[] これはサービスアカウントですか、それとも人間のユーザーですか？
[] トークンはダウンストリームのログ (CDN、WAF、API ゲートウェイ) に表示されていますか？
```

### 短期アクション (1～24 時間)

-**特定のトークンに高い権限が付与されていて、正当なユースケースが確認できない場合は、そのトークンを取り消す**
-**アカウントオーナーにアウトオブバンドチャネル (メール/SMS) で通知** し、認証が開始されたかどうかを確認する
-**SIEM相関のプル**: ソースIPが脅威インテリジェンスフィード、以前に失敗した認証試行、または他のアカウントのログイン履歴に表示されているかどうかを確認します
-**トークンの有効期限ポリシーの確認**: 何時間も使用されずに放置されているトークンは、攻撃対象領域が長引く
