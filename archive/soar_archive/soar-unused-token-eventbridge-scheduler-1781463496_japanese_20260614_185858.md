## イベントコンテキストに関する予備注記

完全な分析に進む前に、重要な観察点を明らかにする必要があります。**提供された JSON コンテキストから、スキャンされたレコードがゼロ、アラートがゼロ、結果もない検出器の呼び出しが明らかになります。** つまり、セキュリティイベントの説明 (ユーザー認証済み、JWT 発行済み、15 分以内に未使用のトークン) は、このシステムが捉えるように設計されている「意図された検出シナリオ」を表していますが、検出器自体には**可視性またはパイプラインギャップ**がある可能性があります。トークンをまったくスキャンできませんでした。このメタファインディングは、個々の未使用のトークンイベントよりも間違いなく重要であり、この分析全体を通じて取り上げられる予定です。

## 1。重要度評価

### 評価:**中** (個別イベント) → **高** (検出器の障害)

| 寸法 | 評価 | ジャスティフィケーション |
|---|---|---|
| 個別の未使用トークン | LOW—MEDIUM | 未使用のトークンだけでは、良性の説明が多い弱いシグナルである |
| 検出器が `scanned: 0`を返す | HIGH | 何もスキャンしない監視システムでは誤った保証が得られる |
| トークンが盗まれた場合の爆発範囲 | 中～高 | トークンの範囲、有効期限、署名アルゴリズムにより異なる |
| 検出ギャップリスク | 高 | 監視が不明であることを知っている攻撃者は自由に活動できる |

### 正当化

未使用のJWTトークンは、**忠実度の低い個別指標**です。ユーザーはセッションを放棄し、ブラウザを閉じると、ネットワーク障害が頻繁に発生します。ただし、他のシグナル (認証情報の乱用、転送不能、以前のログイン失敗) と相関関係があると、攻撃チェーンにおける意味のあるリンクとなります。

より差し迫った懸念事項は、**ディテクタの正常性障害**です。「unused_token_detector」はイベントブリッジスケジューラーによって呼び出されましたが、処理されたレコード数は**ゼロ** (「スキャン済み:0」) でした。これは単なる健康診断書ではなく、**サイレント障害モード**です。アクティブなユーザーがいる実稼働環境で機能するディテクターは、事実上、少なくとも一部のトークン・レコードをスキャンする必要があります。スキャンされたレコードがない場合は、次のパイプライン障害が 1 つ以上発生していると考えられます。

-トークンストア (Redis、DynamoDB、データベース) にアクセスできないか、空の結果が返される
-クエリロジックにバグがある (タイムウィンドウフィルタが間違っている、インデックスが間違っているなど)
-EventBridge ターゲットの呼び出しは成功したが、ダウンストリーム Lambda/サービスは何も通知せずに失敗した
-トークン発行イベントが、ディテクタークエリのストアに書き込まれていません

これにより、**検出の盲点**が生じます。システムは未使用のトークンを監視していると認識していますが、実際には何もしていません。

## 2。考えられる説明を可能性別にランク付け

### 未使用トークンイベントについて

| ランク | 説明 | 発生可能性 | 注意事項 |
|---|---|---|---|
| 1 | ユーザーがセッションを放棄した (タブを閉じて別の場所に移動した) | 非常に高い | 最も一般的な現実的な原因 |
| 2 | クライアント側のエラーによりトークンの使用が妨げられた (CORS、ネットワーク障害、JS エラー) | 高 | SPA とモバイルアプリで一般的 |
| 3 | 自動ヘルスチェックまたは合成モニターは認証されたが処理されない | High | CI/CD パイプライン、アップタイムモニター |
| 4 | 間違った環境でユーザーが認証された (ステージングとプロダクション) | Medium | 開発者/QA エラー |
| 5 | API 呼び出しなしで目的を完了したボット/スクレイパーに発行されたトークン | Medium | 認証情報を検証するためだけに認証が必要なスクレイパーもある |
| 6 | **認証情報検証攻撃** — ダウンストリームアラートをトリガーせずに認証情報が有効であることを攻撃者が確認した | 中程度 | 巧妙な攻撃者は、検出閾値以下を維持するためにトークンを使用することを避ける |
| 7 | **トークン・ハーベスティング** — 攻撃者がMITMまたはXSS経由でトークンを取得し、後で別のホストから使用する | Low—Medium | *この*クライアントがこれを使用しなかった理由を説明する |
| 8 | 競合状態またはクロックスキューにより、トークンは最初に使用する前に期限切れになる | 低い | NTP ドリフトのある分散システムで発生する可能性がある |
| 9 | 内部脅威プロービング認証システム | 低 | ユーザーに特権がある場合は調査が必要 |
###「スキャン済み:0」の検出器障害の場合

| ランク | 説明 | 可能性 |
|---|---|---|
| 1 | タイムウィンドウまたはインデックスの設定が間違っているため、トークンストアクエリが空を返す | High |
| 2 | Lambda/サービスのコールドスタートまたはスキャン完了前のタイムアウト | 高 |
| 3 | 検出器が読み取るストアにトークン発行が書き込まれない | 中—高 |
| 4 | 権限の問題 — ディテクタにトークンストアへの読み取りアクセス権限がない | 中程度 |

| 6 | EventBridge ルールが誤って設定されている — 検出器を起動するが間違ったパラメーターを渡す | 中程度 |
| 7 | インフラストラクチャへのアクセス権を持つ脅威アクターによる意図的な抑制 | 非常に低い |

## 3。アナリストが推奨するアクション

### 即時 (0 ～ 1 時間)

**ステップ 1: 最初に検出器の正常性の問題をトリアージしてください**

```
優先度:重大 — 検出器の故障はシステム上のリスクであり、単一事象のリスクではありません。
```

-EventBridge スケジューラールールが正しく設定され、ターゲット (Lambda ARN、ECS タスクなど) に到達可能であることを確認する
-CloudWatch ログで「unused_token_detector」の Lambda/サービスの実行を確認 — トークンストアクエリからエラー、タイムアウト、または空の結果セットがないか調べます
-ディテクターの実行コンテキスト (VPC、セキュリティグループ、IAM 権限) からトークンストア (Redis、DynamoDB、RDS) にアクセスできることを確認する
-トークンストアに対してディテクタークエリを手動で実行し、トークンが存在するかどうかを確認します

**ステップ 2: 特定の認証イベントを調査する**

-イベントの認証ログをすべて取り出す:
 -ソース IP、ユーザーエージェント、ASN、位置情報
 -認証方法 (パスワード、SSO、MFA)
 -アカウントタイプ (ヒューマンユーザー、サービスアカウント、API キー)
 -このアカウントの以前の認証履歴
-以前に失敗したログイン試行の有無を確認 (クレデンシャルスタッフィングインジケータ)
-ソース IP が脅威インテリジェンスフィードに表示されているかどうかを確認する
-異なる位置からの同時セッションを確認 (移動不可)

**ステップ 3: トークンの露出リスクを評価**

-JWT のクレームを特定する:`サブ`、`スコープ`、`ロール`、`exp`、`aud`
-トークンが機密リソース (管理 API、個人情報、財務データ) へのアクセスを許可しているかどうかを判断する
-トークンがダウンストリームのサービスログに表示されていないか確認する (本当に未使用であったことを確認するため)
-トークンがまだ有効期間内であり、リスクが高まっている場合は、取り消しを検討してください

### 短期 (1～24 時間)

-同じソース IP またはユーザーアカウントについて、過去 24 時間以内に発生したすべての認証イベントを監査する
-トークン発行パイプラインを確認して、ディテクタークエリがストアに書き込まれていることを確認する
-検出器のヘルスチェックを実装します。「スキャン済み == 0」でアクティブなユーザーが存在する場合は、「DETECTOR_HEALTH_FAILURE」アラートを起動します
-同じセッション識別子の WAF ログ、CDN ログ、およびアプリケーションログとの相互関連付け

### 中期 (1～7 日)

-「unused_token_detector」パイプラインの徹底的な監査をエンドツーエンドで実施する
-トークン TTL ポリシーの見直し — 15 分の未使用のしきい値は妥当ですが、セッションアーキテクチャに合っていることを確認してください
-ディテクタ自体の異常検出を可能にするために、ディテクタ実行1回あたりに予想される「スキャン済み」カウントのベースラインを確立する

## 4。エグゼクティブ・サマリー

未使用のJWTトークン（認証情報検証攻撃、トークンの収集、または放棄されたセッションの潜在的な指標）を検出するように設計されたセキュリティ監視システムが、スケジュールされたトリガーによって呼び出されましたが、**処理されたレコードはゼロ**でした。これは**モニタリングパイプラインの障害**であり、意味のある認証異常クラスが見えなくなっています。
