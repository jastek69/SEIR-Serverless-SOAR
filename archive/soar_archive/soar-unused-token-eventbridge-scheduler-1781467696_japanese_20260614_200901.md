### 正当化

中核となる行動シグナル、つまり認証が成功してから監視ウィンドウ内では決して使用されないJWTというシグナルは、**異常ではあるが本質的に悪意があるわけではない**です。ただし、リスク・プロファイルは複数の複合要因によって高まります。

**要因 1 — 永続認証情報としてのトークン**
JWT トークン、特に有効期限の長いトークン (一般的なデフォルト:1 時間、24 時間、またはそれ以上) は、再認証なしに *ネットワークのどの場所*からでも使用できる、ポータブルで署名された資格情報です。未使用のトークンは無効化されたトークンではありません。有効期限が切れるか、明示的に取り消されるまで有効です。侵害されたクライアント、Man-in-the-Browser 攻撃、または悪意のあるブラウザ拡張を通じてトークンが発行され、すぐに流出した場合、攻撃者は有効な認証情報を保持するようになり、被害者のシステムでそれ以上操作する必要がなくなります。

**ファクター 2 — 検出器の死角**
JSON コンテキストには「スキャン済み」: 0` と「結果_合計」: 0` と表示されます。説明されているイベント (認証+未使用のトークン) が実際に発生した場合、ディテクタは次のいずれかになります。
-関連するログソースの取り込みに失敗しました
-パイプラインにギャップがある (イベントブリッジ → ラムダ/プロセッサ → データストア)
-古いデータセットまたは空のデータセットに対して動作している
-しきい値またはフィルターの設定が間違っている

これは、**検出コントロール自体が信頼できない**ことを意味し、検出結果がないことを脅威がないことの証拠として信用できないため、このカテゴリの検出結果の重要度が高まります。

**係数 3 — 爆風半径**
トークンの範囲 (クレーム、役割、権限) によっては、盗まれた JWT が API、マイクロサービス、データストア、または管理機能へのアクセスを許可する可能性があります。トークンの「sub`」、「aud`、`scope`、または「roles」クレームを知らなければ、爆発半径は定義できず、それ自体がリスクとなります。

## 2。考えられる説明を可能性別にランク付け

### ランク 1 — 良性のユーザー行動 (確率:~ 45%)
**説明:** ユーザーが認証され (ログインリンクをクリックした、SSOをトリガーするなど)、トークンを受け取ったが、その後ブラウザタブを閉じた、中断された、接続が切断された、または何らかのアクションを実行する前に別の場所に移動した。

**サポート指標:**
-1 つの認証イベント、以降の API 呼び出しなし
-認証イベントに地理的異常やデバイス異常は発生しない
-ユーザーのタイムゾーンの営業時間中に発生
-ユーザーには同様のショートセッションの履歴がある

**依然として重要な理由:** 無害な未使用のトークンであっても攻撃対象となります。ユーザーのセッションが共有デバイスまたは公共のデバイスで放棄された場合、そのトークンはブラウザーのストレージからアクセスできる可能性があります。

---

### ランク 2 — 自動スクリプトまたはボット認証の失敗 (確率:~ 20%)
**説明:** サービスアカウント、CI/CD パイプライン、または自動化プロセスは正常に認証されたが、設定ミス、ネットワークエラー、またはアプリケーションのバグにより続行できず、トークンが未使用のままになっている。
**サポート指標:**
-認証は営業時間外に行われた
-認証を行うユーザーエージェントがブラウザではない (例:「python-requests」、「curl」、カスタム SDK)
-ソース IP が既知の内部自動化サブネットまたはクラウドプロバイダー CIDR に属している
-アカウントが人の ID ではなくサービスアカウントである

**リスクの影響:** サービスアカウントトークンには、多くの場合、上位の権限が付与されます。トークンを消費せずに認証を繰り返す自動化パイプラインが壊れていると、有効で未使用の権限の高いトークンのプールが生成され、それぞれが攻撃の機会となる可能性があります。

---

### ランク 3 — クレデンシャル・スタッフィング/アカウント・テイクオーバー・プローブ (確率:~ 15%)
**説明:** 攻撃者は有効な認証情報 (侵害、フィッシング、または認証情報の盗用によって入手した) を使用して JWT を認証し、収集しました。攻撃者が別の場所から使用しようとしている、または今後の攻撃に備えて準備中であるため、トークンは*監視対象環境から*使用されませんでした。

**サポート指標:**
-認証ソース IP が VPN 出口ノード、Tor 出口ノード、またはユーザーに関連付けられていないデータセンター IP
-ユーザーにとって通常とは異なるタイミングで認証が行われた。
-ユーザーの認証情報が最近、侵害データセットに表示された（内部脅威情報であるIbeenPwnedを確認）
-認証が成功する前に複数の認証失敗が発生した（認証情報スタッフィングパターン）
-User-Agent 文字列が汎用的であるか、ユーザーの既知のデバイスと一致しない

**攻撃経路の詳細:**
```
攻撃者が認証情報を取得 (侵害/フィッシング)
 ↓
正規のログインエンドポイント経由で認証する
 ↓
JWT の発行と取得 (プロキシ、悪意のある拡張機能、または直接 API 呼び出し経由)
 ↓
攻撃者のインフラにトークンが流出した
 ↓
[サイレンス — 被害者のIPからのアクティビティなし]
 ↓
攻撃者が制御するIPから後で使用されるトークン
 ↓
ラテラルムーブメント、データ漏洩、権限昇格
```

これは特に危険です。なぜなら、トークンの発行とトークンの使用のギャップは数時間から数日に及ぶことがあり、堅牢なトークンレベルのテレメトリなしでは相関関係が困難になるからです。

---

### ランク 4 — クライアント側攻撃によるトークンの流出 (確率:~ 10%)
**説明:** ユーザーは合法的に認証されたが、クライアント側の攻撃 (XSS、悪意のあるブラウザ拡張機能、侵害された JavaScript 依存関係) により、ユーザーが使用する前に JWT が傍受され、漏洩した。

**サポート指標:**
-アプリケーションは JWT を「ローカルストレージ」または「セッションストレージ」に保存します (JavaScript でアクセス可能)
-フロントエンドの依存関係に関する最近の変化 (サプライチェーンリスク)
-ほぼ同時期に記録された CSP (コンテンツセキュリティポリシー) 違反
-ログイン後にアプリケーションが「動作しなかった」とユーザーが報告した

**これが陰湿な理由:** 正規のユーザーが通常のログインフローを経験した可能性があります。トークンの盗難はサーバー側のロギングからは見えません。唯一確認できたのは、説明したとおりの結果だけです。認証が成功し、トークンが発行され、想定していたクライアントからトークンが使用されなかったということです。

---

### ランク 5 — 検出器/パイプラインの誤動作 (確率:~ 8%)
**説明:** 説明したイベントが説明どおりに発生しなかったか、ディテクタが誤ったロジックに基づいてファントムアラートを生成している可能性があります。JSON の「scanned」: 0` はこれを要因として強く裏付けています。

**サポート指標:**
-`"scanned」: 0` — 実際に評価されたレコードはありません
-「結果合計」: 0` — 処理中のデータがない場合でも一貫性があります
-トリガーが起動された（EventBridge スケジューラーが実行された）が、ダウンストリームプロセッサは分析対象を見つけられなかった
-考えられる原因:データソースが空である、データ読み取りを妨げている IAM 権限エラー、Lambda コールドスタートタイムアウト、タイムウィンドウクエリが正しくない

---

### ランク 6 — 内部脅威/意図的なトークンハーベスティング (確率:~ 2%)
**説明:** 悪意のある内部ユーザーまたは侵害を受けた特権ユーザーが、後で使用するためのトークンを生成したり、外部の第三者と認証情報を共有したりするために、意図的に認証を行っています。

**サポート指標:**
-認証アカウントが属している
