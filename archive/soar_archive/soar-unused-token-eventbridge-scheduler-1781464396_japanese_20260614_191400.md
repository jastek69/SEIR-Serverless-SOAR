### ジャスティフィケーション・ナラティブ

前述のシナリオ、つまり、ユーザーが認証されて JWT を受け取ったのにそれを使用しないというシナリオは、**無害なユーザー行動**と**認証情報を収集するトレードクラフト**が交わる興味深いシナリオです。単独では、未使用のトークンが 1 つでも重要度は低くなります。ただし、「scanned: 0」と報告されたディテクターは、このパターンが大規模に発生しているかどうかを組織が現在**把握できていない**ことを意味します。クレデンシャル・スタッフィング、フィッシングによるセッション・トークン・ハーベスティング、またはOAuthの悪用を行う脅威アクターは、まさにこのシグネチャを生成します。有効な認証の後にトークンを発行し、トークンを取り出して後で使用または売却するために保管するというものです。これを大規模に検出できないため、「検出ギャップ」の深刻度は「高」に高まります。

## 2。考えられる説明を可能性別にランク付け

### ランク 1 — ディテクタの設定ミスまたはデータソースの障害* (発生率:非常に高い) *

「scanned: 0」という結果が、最もすぐに行動に移せる結果です。考えられる原因:

-EventBridge Scheduler は Lambda/関数を正しくトリガーしたが、ダウンストリームクエリ (DynamoDB、Redis、RDS など) はクエリの設定ミス、テーブル/インデックスの誤り、タイムウィンドウフィルタの誤り、またはデータソースの IAM 権限拒否により空の結果セットを返した。
-トークンストアに、すべてのレコードを除外するフィルタを使用してクエリを実行しました（例：「WHERE issued_at < NOW ()-5 分間隔」でタイムゾーンが一致しない）。
-検出器が最近導入されたため、トークン発行イベントがスキャン対象とは別のデータストアに書き込まれます。
-サイレント例外がキャッチされて飲み込まれたため、関数はエラーを発生させずに何も結果を出さずに正常に終了しました。

**これが最初にランクされる理由:** JSON は明確です。システムが新品だったり、データパイプラインが壊れていたりしない限り、アクティブなユーザーがいる実稼働環境でゼロスキャンを行うことは統計的に不可能です。

---

### ランク 2 — 良性のユーザー行動 (放棄されたセッション) * (発生率:高 — 前述のシナリオでは) *

ユーザーが認証 (「ログイン」ボタンをクリックするなど) して JWT を受け取り、次の操作を行います。

-ログイン後すぐにブラウザタブを閉じます。
-クライアント側の JavaScript エラーが発生し、トークンの保存や使用ができなくなる。
-SPA (シングルページアプリケーション) が初期化を完了する前に移動する。
-最初の認証済み API 呼び出しを行う前にクラッシュしたモバイルアプリ経由で認証する。
-事前認証はするが実際の使用は保留する「Remember me」フローを使用する。
これは、個々の未使用のトークンイベントに関する最も一般的な現実的な説明であり、**セキュリティ上の脅威はありません**を表しています。

---

### ランク 3 — 自動認証情報検証/スタッフィング * (可能性:中) *

クレデンシャルスタッフィング操作を頻繁に実行する脅威アクター:

1.盗んだユーザー名とパスワードのペアを認証エンドポイントに送信します。
2.生成された JWT を収集して、認証情報が有効であることを **検証**します。
3.トークン (または検証済みの認証情報のみ) を後で使用または販売できるように保管します。
4.アプリケーションのビジネスロジックエンドポイントに対してトークンを実際に使用しないでください。

これにより、「認証が成功した」→「JWT が発行された」→「トークンが使用されていない」という説明どおりのシグネチャが生成されます。規模が大きくなると、このパターンは自動認証情報検証の有力な指標となります。攻撃者の目的は認証情報のインベントリであり、即時のアカウント乗っ取りではありません。

**攻撃経路の詳細:**
```
攻撃者 → 認証エンドポイント (認証情報がぎっしり詰まった状態の POST /ログイン)
 → 200 OK + JWT を受け取る
 → 認証情報を「有効」として記録する
 → 次の認証情報ペアに移動
 → JWT は未使用で期限切れになる
```

**検出ギャップ:** 未使用のトークン検出機能がなければ、認証エンドポイントは有効な認証情報ごとに 200 を返すため、このキャンペーンはアプリケーションログにまったく表示されません。正規のログインと区別がつかないためです。

---

### ランク 4 — フィッシング/トークンハーベスティング * (可能性:中程度) *

ユーザーは次のような説得力のあるフィッシングページに誘導されます。

1.実際の認証フロー (Evilginx2 スタイルなど、中間層の攻撃者) をプロキシします。
2.送信中に発行された JWT をキャプチャします。
3.トークンのないエラーページまたは実際のアプリケーションにユーザーをリダイレクトします。

正当なユーザーはトークンを使用しません (エラーが表示されたり、再認証されたりします)。攻撃者は後で使用できるようにトークンを保持します。未使用のトークンウィンドウ (5 ～ 15 分) は、収集したトークンを使用する前に攻撃者が操作を遅らせている可能性があります。

---

### ランク 5 — サービスアカウント/API クライアントの設定ミス* (発生率:中～低) *

自動化されたサービスまたは CI/CD パイプライン:

-ヘルスチェックまたはスモークテストの一環として JWT を取得することを認証します。
-バグ、ネットワークタイムアウト、または設定エラーにより、次のステップに進めない。
-トークンは未使用のままにします。

これは問題ありませんが、オートメーションパイプラインが壊れていることを示しているので、別途調査する必要があります。

---

### ランク 6 — 偵察/アプリケーションフィンガープリンティング* (可能性:低) *

アプリケーションを調査している攻撃者が、テストアカウントまたは侵害されたアカウントで次の目的で認証を受ける可能性があります。

-JWT をデコードしてクレーム (発行者、対象者、アルゴリズム、カスタムクレーム、ユーザーロール) を抽出します。
-アプリケーションの認証アーキテクチャをマッピングします。
-トークンの有効期限や署名アルゴリズム (特に `alg: none` や弱いアルゴリズムが使用されている場合) を特定する。

トークンはビジネスロジックに対して使用されることはなく、デコードされて分析されるだけです。

---

### ランク 7 — 内部脅威/権限昇格ステージング* (発生率:低) *

悪意のある内部関係者は、セカンダリアカウントまたはサービスアカウントを使用して次のことを認証します。

-上位権限のトークンを取得する。
-特定の時間 (営業時間外、メンテナンス時間中など) に使用できるようにステージングする。
-リアルタイム監視を回避するためにすぐに使用することは避けてください。
