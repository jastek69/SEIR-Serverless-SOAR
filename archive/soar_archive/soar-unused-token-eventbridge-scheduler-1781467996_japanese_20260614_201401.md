## 暫定メモ:コンテキスト調整

先に進む前に、重要な所見を明らかにしておく必要があります。**説明イベントの記述と JSON コンテキストペイロードには重大な相違があります。** 説明文では特定のユーザーイベント (認証 → JWT 発行 → 15 分以内にトークンが未使用) が記述されていますが、JSON テレメトリーでは次のことがわかります。

-`scanned: 0` — 実際に評価されたトークンはない
-`alerted: 0` — アラートは生成されませんでした
-`結果_total: 0` — 結果はありません
-`ファインディングス_サンプル:[] `— 空の結果セット
-`閾値_分:5` — 検出器の閾値は 15 分ではなく 5 分です

**つまり、解析では次の 2 つの異なる懸念事項に同時に対処する必要があるということです。**

1.**仮説的/意図されたシナリオ** — もし実際の発見があったら、未使用のトークン・イベントはどういう意味になるのか
2.**実際の運用上の懸念** — 検出器自体が誤動作しているか、正しく構成されていないように見える。これ自体がセキュリティ上のリスクである

両方のスレッドを以下で分析します。

## 1。重要度評価

### シナリオ A: 未使用のトークンが実際に検出された場合
**重要度:中 (高へのエスカレーションパスあり) **

| 要因 | 評価 |
|---|---|
| 機密保持への影響 | 中 — トークンは有効な認証期間を表す |
| インテグリティ・インパクト | 低・中 — トークンの範囲/クレームにより異なる |
| 可用性への影響 | 低 |
| 悪用可能性 | 中 — トークンの傍受が必要 |
| Blast Radius | トークンの権限セットに限定される。権限があれば広範囲に及ぶ可能性がある |

**理由:**
発行は成功したが一度も使用されていない JWT は、それ自体が異常です。通常のユーザー行動では、認証後ほぼ即時にトークンが使用されます。発行と未使用のギャップは、いくつかの脅威条件のうちの 1 つを示唆しています。以下の場合、深刻度は大幅に高まります。
-トークンには上位の権限 (管理、書き込み、財務スコープ) が付与されている
-ユーザーアカウントがサービスアカウントまたは共有認証情報である
-ソース IP が異常であるか、地理的に一貫性がない
-同じユーザーまたは IP 範囲にこのようなイベントが複数集まっている

### シナリオ B: スキャン回数がゼロの検出器 (実際のイベント)
**重要度:中～高**

**理由:**
起動するが何もスキャンしないセキュリティコントロールは、**検出ギャップ**であり、完全な状態を示すものではありません。これは、作動しているが空気をサンプリングしない煙感知器と動作的には同等です。ここでのリスクは以下のとおりです。
-実際の未使用のトークンの悪用が、**可視性がゼロ**の状態で発生している可能性があります
-EventBridge Scheduler がディテクターを呼び出し、パイプラインが動作していることを確認しているのに、ディテクターロジックが通知なしで失敗している
-これにより、最も危険なセキュリティ態勢である**誤った安心感**が生まれます。

## 2。考えられる説明を可能性別にランク付け

### 検出器の誤動作について (`スキャン済み:0`)
| ランク | 説明 | 可能性 | 証拠 |
|---|---|---|---|
| 1 | **トークンストアクエリが空のセットを返す** — ディテクターは、空のトークン、設定ミス、または間違った環境/名前空間を指しているトークンのレジストリ/キャッシュを照会します。| Very High | 機能しているスケジューラで `scanned: 0` が発生すると、データソースの問題が発生する可能性が強くなります |
| 2 | **環境設定ミス** — Detectorが間違った環境で実行されている（例：dev detectorがprodトークンのストアパスを誤ってスキャンしている、またはその逆）| 高 | 共有EventBridgeルールを使用するマルチ環境デプロイメントで一般的 |
| 3 | **トークンのインデックスパイプラインが壊れています** — トークンが発行されているが、ディテクターが読み取るストアに書き込まれていない (Redis TTL が短すぎる、DynamoDB の書き込み失敗、非同期書き込みの欠落など) | 高 | スキャンがゼロと結果ゼロの両方を説明できる |
| 4 | **IAM/権限リグレッション** — ディテクターを実行している Lambda/サービスが、最近のデプロイ後にトークンストアへの読み取り権限を失った | Medium | 通常はエラーを生成するが、SDK 構成によってはサイレント障害が発生する可能性がある |
| 5 | **デプロイ後のディテクターロジックのバグ** — 最近のコード変更により、スキャンループが実行される前にショートサーキットリターンが発生しました。| Medium | 最近のデプロイの相関関係が必要 |
| 6 | **攻撃者による意図的な抑制** — 検出パイプラインへのアクセスを無効にした、または検出されずに動作するようにスキャナを去勢した攻撃者 | Low-Medium | Plassiful; 事前にかなりのアクセスが必要だが、無視してはいけない |

| ランク | 説明 | 可能性 | エビデンス指標 |
|---|---|---|---|
| 1 | **ユーザーがセッションを放棄しました** — API 呼び出しを行う前に認証され、中断され、ブラウザ/アプリケーションが閉じられました | 非常に高い | 相関する API 呼び出しなし、単一認証イベント、通常の営業時間 |

| 7 | **セッション固定または送信中のトークンの盗難** — トークンは発行後に傍受された。攻撃者はトークンを持っているが、正当なユーザーはそれを受け取っていない | 低中程度 | MITM機能が必要である。TLSの設定ミス、エンドポイントの侵害 |
